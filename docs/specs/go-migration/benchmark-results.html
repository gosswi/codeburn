Key findings

Go mattn warm: 22-44x faster than installed

Per-file SQLite caching reads 180 pre-aggregated session summaries in 52-58ms. This is a 7-8x improvement over TS Phase 2 warm (370-410ms). The Go binary parses no JSONL on a warm run — it reads directly from the SQLite cache and aggregates in memory. RSS drops to 31MB (vs 284MB installed) because no session data stays in memory.

Cold start: Go is already 3-6x faster than installed

Even with an empty cache, Go mattn parses all 180 sessions cold in 375-475ms vs 1122-2563ms for installed. No Node.js startup, no V8 JIT warmup. JSONL parsing in Go is roughly 3x faster. TS Phase 2 cold (773-802ms) is also faster than installed but slower than Go cold due to Node.js overhead.

Why TS Phase 2 warm (370ms) lags Go warm (52ms)

Both TS Phase 2 and Go mattn use per-file SQLite caching with the same fingerprint strategy (mtime + file size). The gap exists because Node.js adds ~300ms of startup and V8 initialization overhead on every invocation. Even when reading from cache, TS still needs to load the JS runtime, parse the bundle, and initialize the SQLite connection. Go has no such overhead.

Cache isolation note: shared SQLite DB

Both TS Phase 2 and Go mattn write to ~/.cache/codeburn/session-cache.db with the same schema. However, their JSON formats are incompatible: Go uses PascalCase field names, TS uses camelCase. A resilience fix was applied to TS Phase 2 to detect and evict Go-format rows rather than crashing. Long-term, adding json:"..." struct tags to Go types would make caches portable.
