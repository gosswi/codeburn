Why end-to-end is unchanged

The session cache stores summaries for non-Claude providers (Cursor, Codex). Claude directory sources bypass the SQLite cache (mtime+size fingerprinting is not reliable on directories). Since Claude is the dominant workload (173+ session files), the bottleneck is JSONL parsing, not SQLite reads.

The 1.78x mattn speedup applies to every SQLite read/write, but those operations account for only a small fraction of the total 730ms. The benefit will compound if Claude directory caching is added in a future spec.

Key findings

mattn vs modernc: SQLite speedup, null end-to-end

The mattn driver is 1.78x faster for SQLite operations (47K vs 84K ns/op). However, end-to-end wall-clock is unchanged (~730ms both). The session cache skips Claude directory sources -- the dominant workload. SQLite is not the bottleneck.

Real benefit: RSS drops 10-15% on darwin

mattn's native C library eliminates the pure-Go runtime overhead that modernc carries. Result: 88-95MB RSS with mattn vs 104-107MB with modernc -- matching or beating TS Phase 2's 98-101MB. Linux binaries retain modernc (CGO_ENABLED=0) for static distribution.

Next bottleneck: Claude directory caching

The gap between Go (~730ms) and TS Phase 2 (~370ms) is caused by Claude JSONL re-parsing on every invocation. Adding SQLite caching for Claude directory sources would allow the mattn speedup to compound -- reading pre-parsed summaries in 47K ns/op vs full JSONL parse. This is the next actionable optimization.

Why TS Phase 2 still leads Go by ~2x

better-sqlite3 (native C, synchronous) caches all 173+ session summaries and returns them in a single fast read. Go re-parses every Claude JSONL file on each invocation because directory-level mtime fingerprinting is unreliable. The fix is file-level caching per JSONL file within Claude project directories.
